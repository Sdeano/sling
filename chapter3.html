Step 3 - Set up a new model that stores the global application state

With the approval functionality, our application needs to handle different states. For example, once we call the business logic in the backend, we need to ensure that the purchase order instance is locked to prevent the user from clicking the approve button again. We also need a defaulting functionality to select the next purchase order once a purchase order is approved and removed from the list, this includes triggering a re-fresh of the list and detail view. Since part of the information is bound to elements in different views, a model is the best place to store the relevant information. Our application contains already four models: 1. OData service that encapsulates the business logic 2. Resource bundle for translatable texts 3. Device 4. FLP The first two are taken from the application descriptor manifest.json whereas the latter two are defined in model/model.js. The code to manage these models was generated for us from the template wizard. We’ll use the same mechanics for our new model to discuss what the template wizard already did for us and to get a consistent model usage within our app. So let’s define our new model in model/model.js too:


sap.ui.define([
"sap/ui/model/json/JSONModel",
"sap/ui/Device",
"sap/ui/model/BindingMode"
], function (JSONModel, Device, BindingMode) {
"use strict";

return {
[…]
createGlobalModel : function() {
var oModel= new JSONModel({
isBusyApproving: false,
isSwipeRunning: false
});
oModel.setDefaultBindingMode(BindingMode.TwoWay);
return oModel;
}
};
}
);

In contrast to the FLP and device model, this model requires two way binding. This is because FLP and de-vice model properties will not change during the execution of our app whereas the new model depends on different states. Furthermore, since we’ll change values via DataBinding and programmatically via setProper-ty, a two way binding in necessary. We’ll discuss the properties of this model later on.
